Interviu Pasiruošimas – Povilas Urbonas

1. Trumpas prisistatymas

Esu front-end programuotojas su daugiau nei 8 metų patirtimi kuriant interaktyvias, patikimas ir vartotojui draugiškas sistemas, daugiausiai naudojant ReactJS, JavaScript ir TypeScript. Esu dirbęs tiek front-end, tiek full-stack pozicijose, todėl puikiai suprantu visą programavimo ciklą. Šiuo metu vadovauju mažai komandai IT Sprendimai MB, kur ne tik programuojame WordPress sistemas su PHP ir JS, bet ir diegiame API, debesijos ir techninio efektyvumo sprendimus. Anksčiau dirbau Allente, kur prisidėjau prie didelio masto platformos, taip pat Tribecoje, kur kūriau vidaus įrankius .NET ir Blazor pagrindu.

2. Projektų aptarimas

Projektas 1 – Allente (React + TypeScript + ASP.NET)

Dirbau prie klientų savitarnos portalų, kur pagrindinis tikslas buvo suteikti sklandžią naudotojo patirtį, įskaitant prisijungimą, užsakymų valdymą, klientų aptarnavimą. Vienas svarbiausių indėlių buvo Webpack konfigūracijos optimizavimas – sutrumpinome „bundle“ dydį ~35%, pagerinome puslapio įkrovimo greitį. Taip pat įdiegėme Adobe Analytics stebėjimą, leidusį komandai priimti geresnius sprendimus apie naudotojo elgseną.

Projektas 2 – Tribeca (ASP.NET + Blazor + React)

Kūriau vidaus įrankius, skirtus dokumentų valdymui ir ataskaitoms. Vienas įdomesnių iššūkių – automatizuoti ataskaitų generavimą į MS Word dokumentus iš SQL duomenų bazės. Sukūriau generavimo įrankį naudojant C# ir Word API – tai sumažino rankinių procesų poreikį nuo valandų iki sekundžių.

3. Techniniai klausimai

  • React Hooks: Naudoju useState, useEffect, useMemo ir useCallback. Pvz., Allente projekte naudojau useEffect duomenų sinchronizacijai tarp komponentų ir API.
  • TypeScript: Pasirinktas dėl geresnio „developer experience“, automatinio klaidų aptikimo ir didesnio saugumo dideliuose projektuose.
  • State valdymas: Naudoju Context API ir Redux Toolkit, priklausomai nuo projekto sudėtingumo. Konkrečiai WordPress projekte naudojome Context UI režimo sinchronizavimui.
  • REST API: Teko tiek kurti (su Laravel ir .NET), tiek naudoti. Naudoju axios ar fetch, taip pat įgyvendinau klaidų valdymą su retry mechanizmais.
  • Optimizavimas: Code splitting su React.lazy, lazy loading komponentai, dinaminis importas, suspaudimas, `image/webp` naudojimas.
  • Kodo kokybė: ESLint, Prettier, GitHub Actions CI, code review kultūra, kartais Snapshot testai su Jest.

4. Klausimai darbdaviui

  • Kaip atrodo komandos struktūra? Kiek žmonių dirba prie front-end?
  • Kaip priimami technologiniai sprendimai? Ar turėsiu įtakos architektūrai?
  • Ar naudojate CI/CD, kokią testavimo strategiją taikote?
  • Ar darbuotojams suteikiamos galimybės mokytis, tobulėti technologiškai?

5. Mano lūkesčiai

Noriu dirbti komandoje, kur skiriamas dėmesys kokybiškam kodui, kur galiu augti kaip React specialistas, bet taip pat prisidėti prie architektūros ir full-stack sprendimų. Domina projektai, kuriuose galima naudoti naujas technologijas – pvz. Next.js, Vite, Server Components. Tikiu stipria bendravimo kultūra ir noriu dirbti su žmonėmis, kurie rūpinasi galutiniu produktu.

6. Klausimas: Papasakok apie problemą, kurią sprendei, ir kaip tai pavyko

Klausimas: Ar gali papasakoti apie realią problemą, su kuria susidūrei projekte, ir kaip ją išsprendei?

Atsakymas:
Allente projekte turėjome problemą, kai kai kuriuose įrenginiuose naudotojo sąsaja kraudavosi labai lėtai. Išanalizavus su Lighthouse paaiškėjo, kad mūsų „bundle.js“ failas siekia per 1.5MB. Inicijavau veiksmų planą – atlikome lazy loading modulių, išskirstėme trečiųjų šalių bibliotekas, pašalinome nenaudojamus komponentus ir atnaujinome Webpack konfigūraciją. Rezultate pasiekėme 40% greitesnį puslapio įkėlimą ir pagerinome „performance score“ nuo 55 iki 88.
